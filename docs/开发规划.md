## 二、三大主线工作

### 主线 A：AI 助手核心功能完善

**目标：把现有半成品补完，让 AI 助手"能用且好用"。**

#### A1. 天气 GPS 自动定位（C 岗 / 兰协助）

**现状问题：** `environment_agent.py` 第 40 行 `DEFAULT_CITY = "Beijing"`，无定位逻辑。

**实施步骤：**
1. 在 `EnvironmentAgent` 增加 `get_gps_location()` 方法，调用设备 GPS / 网络定位 API
2. 用 reverse geocoding（高德/百度地图 API）将经纬度转城市名
3. 改造 `_fetch_weather()` 入参链路：`{gps: (lat, lon)}` → `city` → 天气查询
4. 前端 `NavPanel.vue` 增加定位状态指示器（定位中 / 已定位 / 定位失败）

**交付标准：**
- 启动后自动获取设备位置 → 反查城市 → 拉取天气
- 定位失败时降级回 `DEFAULT_CITY` 并提示用户
- 前端显示当前城市名和定位来源

**关键技术选型：**
- Web 端：`navigator.geolocation` API（浏览器原生）
- 移动端：uni-app `uni.getLocation()`（HBuilder 原生定位）
- 反查城市：高德 Web 服务 API（免费额度足够实训用）

#### A2. 疲劳趋势预测补完（A 岗）

**现状：** `fatigue_predictor.py` 只有类定义和滑动窗口框架，趋势计算逻辑空缺。

**实施步骤：**
1. 定义 7 维状态向量：`[PERCLOS, 眨眼频率, 平均睁眼时长, 头部yaw方差, 头部pitch方差, 视线偏移频率, 告警响应延迟]`
2. 实现滑动窗口（建议 60 秒）内的特征聚合
3. 实现趋势计算：线性回归斜率 + 近期窗口加权
4. 输出 `{fatigue_trend: "rising"/"stable"/"declining", fatigue_score: 0-100, prediction: "likely_drowsy_in_Xmin"}`
5. 接入 `safety_agent.py` 融合判定

**交付标准：**
- 连续驾驶 60 秒后能输出疲劳趋势
- 趋势 + 当前 PERCLOS 联动触发告警升级
- 有自测用例覆盖正常 / 上升 / 危险三种走势

#### A3. 前端告警和仪表盘可视化（A 岗）

**现状：** `AlertPanel.vue` 和 `GaugePanel.vue` 都是空壳。

**实施步骤：**
1. `GaugePanel.vue`：用 ECharts 画 PERCLOS 仪表盘 + 疲劳分数进度条 + 头部姿态 3D 指示
2. `AlertPanel.vue`：三级告警动画（mild 黄色闪烁 / moderate 橙色脉冲 / severe 红色全屏遮罩）+ 与 TTS 语音联动
3. 整合到 `DashboardView.vue` master 框架

**交付标准：**
- 后端推送 `risk_level` 变化时，前端 100ms 内响应
- 三级告警动画流畅，不卡顿
- 仪表盘数据实时更新

#### A4. Whisper 实时语音接入（组长）

**现状：** `speech_recognizer.py` 模块完整，未接入 `app.py` 实时流程。

**实施步骤：**
1. 在 `app.py` 主循环中并行启动音频采集线程
2. 配置 VAD（语音活动检测）触发转写，避免持续录音
3. 转写结果送入 `local_decision_engine` 关键词匹配 → 命中直接执行
4. 未命中关键词 → 送 `InteractionAgent` 走 DeepSeek 语义理解

**交付标准：**
- 说"打开空调"等关键词指令，500ms 内响应
- 说"今天适合开车吗"等复杂语义，走云端 LLM
- 静音时不触发转写，不浪费算力

### 主线 B：移动端迁移（兰主负责）

**目标：Vue3 网页版 → HBuilder/uni-app 安卓原生应用。**

#### B1. 迁移可行性评估与方案定稿（第1阶段）

**现状：** 代码中无任何 HBuilder / uni-app 实现。

**待评估问题：**
1. **前端框架选型**：uni-app 基于 Vue，理论上 Vue3 组件可迁移；但 Element Plus 在移动端不适用，需换 uni-ui 或 uView
2. **摄像头权限**：网页版用 `getUserMedia`，uni-app 需用 `plus.camera` 或 `uni.createCameraContext`
3. **WebSocket**：uni-app 原生支持 `uni.connectSocket()`
4. **MediaPipe 模型**：.task 模型文件需要在移动端重新验证加载路径
5. **后端部署**：Python FastAPI 部署在哪？手机本地跑不通，需要云服务器或局域网 PC

**交付物：**
- 一份迁移方案文档（可以是 md），明确：
  - 哪些组件可直接迁移
  - 哪些必须重写（如 Element Plus 组件 → uni-ui）
  - 哪些功能在移动端不可用或需替代方案
  - 后端部署方案（云服务器 / 局域网 / 本地）

**兰下周一/二向团队同步的内容就是这个。**

#### B2. HBuilder 工程初始化（第2阶段）

**实施步骤：**
1. 用 HBuilderX 创建 uni-app Vue3 项目
2. 配置 `manifest.json`：申请摄像头、麦克风、定位、网络权限
3. 搭建页面路由：主页（大屏）/ 设置页 / 日志页
4. 迁移 Vue3 组件骨架（App.vue / router 逻辑 → pages.json）

**交付标准：**
- HBuilder 能编译出 APK，在安卓手机上能装能开
- 主页能看到基础布局（即使数据是假的）

#### B3. 核心组件迁移（第3阶段）

按优先级迁移：

| 优先级 | 组件 | 迁移难点 |
|--------|------|---------|
| P0 | `DashboardView.vue` → 主页 | 布局重写，Element Plus → uni-ui |
| P0 | `NavPanel.vue` → 天气/导航组件 | GPS 用 `uni.getLocation()` |
| P1 | `AiPanel.vue` → AI 状态面板 | 简单，直接迁移 |
| P1 | `AlertPanel.vue` → 告警组件 | 动画用 CSS，可迁移 |
| P2 | `VoiceInteractionBar.vue` → 语音交互栏 | 录音用 `uni.getRecorderManager()` |
| P2 | `GaugePanel.vue` → 仪表盘 | ECharts 用 `renderjs` 方案 |
| P3 | `AiDecisionPanel.vue` → AI 决策可视化 | 560 行复杂组件，可能精简 |

**交付标准：**
- P0 组件迁移完成，能展示实时数据
- 摄像头能在手机上打开并推流到后端

#### B4. 移动端 GPS 与定位整合（第4阶段）

**与主线 A1 联动。** 移动端定位方案：
- `uni.getLocation({type: 'gcj02'})` 获取经纬度
- 调用高德 Web 服务 API 反查城市
- 传给后端 `EnvironmentAgent`

## 五、对兰的具体建议

你现在权限是 view（presenter），不是代码提交者。这意味着：

1. **迁移代码你只能在本地开发**，成品后需要组长帮忙合入，或者你用自己的 Git 仓库
2. **下周一/二同步时**，建议准备：
   - 迁移方案文档（B1 阶段交付物）
   - 你评估的技术难点和不确定项
   - 你需要的支持（比如组长配合改后端 API、A 岗补完 fatigue_predictor 的时间点）
3. **答辩材料你主负责**，但技术内容需要组长提供准确的架构说明和创新点描述
4. **建议先用 HBuilderX 跑通一个空的 uni-app Vue3 项目**，确认环境 OK 再谈迁移
